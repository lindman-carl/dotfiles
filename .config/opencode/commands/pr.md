# Create Pull Request

Create a GitHub pull request for the currently checked-out branch.

---

## STEP 1: PRE-FLIGHT CHECKS

```bash
# Confirm current branch and ensure it's not the default branch
git branch --show-current
git remote -v
# Check for uncommitted changes
git status --porcelain
```

**STOP if there are uncommitted changes.** Run `/commit` first.

**STOP if on the default branch** (`main`, `master`, `develop`). PRs must come from a feature branch.

---

## STEP 2: DETERMINE BASE BRANCH

```bash
# Detect the default branch of the remote
gh repo view --json defaultBranchRef --jq '.defaultBranchRef.name'
```

Use the detected default branch as the PR base. If the command fails, default to `main`.

---

## STEP 3: PUSH BRANCH TO REMOTE

```bash
# Push the current branch, setting upstream if needed
git push -u origin $(git branch --show-current)
```

---

## STEP 4: GATHER CONTEXT FOR PR CONTENT

Collect all the information needed to write a great PR description.

```bash
# Commits on this branch that are not on the base branch
git log --oneline origin/<base-branch>..HEAD

# Full diff stat for a high-level overview
git diff --stat origin/<base-branch>...HEAD

# Check if any commit refs a GitHub issue
git log --format="%B" origin/<base-branch>..HEAD | grep -Eo "(Closes|Fixes|Refs) #[0-9]+"
```

Also read the changed files for additional context where needed.

---

## STEP 5: WRITE THE PR

Using the gathered context, compose:

### Title

- Imperative mood, max 72 characters
- Mirrors the most significant conventional commit on the branch, or summarises all commits
- Format: `<type>(<scope>): <what this PR does>` (drop the type prefix if the repo does not use conventional commits)
- Examples:
  - `feat(auth): add token refresh endpoint`
  - `fix(dashboard): correct chart data aggregation`

### Description

Structure the body using this template — adapt or omit sections that do not apply:

```markdown
## Summary

<!-- One or two sentences explaining what this PR does and why. -->

## Changes

<!-- Bullet list of the notable changes. Group by concern if many. -->
-
-

## Testing

<!-- How was this tested? Unit tests, manual testing steps, etc. -->
-

## Related Issues

<!-- Issue references, if any. Use "Closes #N" to auto-close on merge. -->
```

**Guidelines:**

- Write for a reviewer who is unfamiliar with the day-to-day context
- Explain *why*, not just *what* — the diff already shows what changed
- Keep bullets short and scannable
- Do NOT add AI attribution or "Generated by" footers

---

## STEP 6: CREATE THE PR

```bash
gh pr create \
  --base <base-branch> \
  --title "<title>" \
  --body "<description>" \
  --draft
```

Always create as a draft unless the user explicitly asks for a ready-for-review PR (omit `--draft` only in that case).

---

## STEP 7: CONFIRM

```bash
# Print the URL of the newly created PR
gh pr view --web
```

Report the PR URL and title to the user.
