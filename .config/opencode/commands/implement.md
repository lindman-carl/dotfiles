
## GIT COMMIT STRATEGY

You MUST create local git commits at the end of the implementation. Commits are a primary artifact of this process — they document not just what changed but why, and trace the progression of decisions so a developer reading the history understands the reasoning at each step.

### Commit Rules

- **Commit after each logical unit of work** — a completed function, a new file, a wired-up integration, a batch of tests
- **Never let more than ~30 minutes of work go uncommitted**
- **Each commit must compile/parse** — do not commit broken code
- **Use Commitizen conventional commit format**:

```text
<type>(<scope>): <subject>

<body>

<footer>
```

- **Keep commits focused** — one concern per commit. If you changed tests AND source, consider two commits
- **Do NOT add AI attribution, co-author tags, or "Generated by" messages**

### Commit Message Structure

Every commit should tell a story. Use the three-part structure:

**Subject line** — imperative mood, max 72 chars, describes *what* changed:

```text
feat(auth): add token refresh service
```

**Body** — explains *why* this approach was chosen, what alternatives were considered, and any non-obvious design decisions. Wrap at 72 chars. This is the most important part for future readers:

```text
Chose short-lived access tokens with refresh rotation over long-lived
tokens to limit the blast radius of token leaks. Considered JWT
stateless approach but rejected it because we need immediate revocation
support for the admin panel.
```

**Footer** — issue references and breaking change notices:

```text
Refs #42
```

or for the final commit that closes the issue:

```text
Closes #42
```

### When to Commit

| After completing…                          | Example commit                                         |
|--------------------------------------------|--------------------------------------------------------|
| A new module/file with real logic          | `feat(auth): add token refresh service`                |
| Wiring up an integration point             | `feat(api): connect auth middleware to router`         |
| A batch of related tests                   | `test(auth): add token expiry edge case tests`         |
| A refactor or extraction                   | `refactor(utils): extract date formatting helpers`     |
| A bug fix discovered during implementation | `fix(auth): handle expired refresh token gracefully`   |
| Verification fixes (lint, type errors)     | `fix(lint): resolve eslint warnings in auth module`    |

### How to Commit

Standard commit (subject only, for obvious/self-explanatory changes):

```bash
git add -A
git commit -m "<type>(<scope>): <subject>"
```

Full commit with body and issue reference (preferred for all non-trivial changes):

```bash
git add -A
git commit -m "<type>(<scope>): <subject>" \
  -m "<body explaining the why and design decisions>" \
  -m "Refs $ISSUE_REF"
```

Closing commit (final commit that resolves the issue):

```bash
git add -A
git commit -m "<type>(<scope>): <subject>" \
  -m "<body>" \
  -m "Closes $ISSUE_REF"
```

### What Makes a Good Commit Body

Think of each commit body as a note to a future developer (or yourself in six months) who is reading `git log` to understand a decision. Answer these questions when relevant:

- **Why** was this approach chosen over the obvious alternative?
- **What constraint** drove this particular design?
- **What was rejected** and why?
- **What tradeoff** was made deliberately?
- **What context** from the issue influenced this?

Bad body: `"Added retry logic"`
Good body: `"Retries use exponential backoff with jitter to avoid thundering herd when the upstream rate limiter resets. Fixed 3 retries based on issue #42 requirement for < 2s p99 latency under normal conditions."`

## PHASE 3: FULL IMPLEMENTATION

Now implement the feature. Follow these ABSOLUTE RULES:

### FORBIDDEN - Never do any of these

- `// TODO:` or `// FIXME:` comments
- `// NOTE:` or `// HACK:` comments about future work
- `throw new Error("Not implemented")` or similar
- `pass` statements as placeholders (Python)
- `unimplemented!()` or `todo!()` macros (Rust)
- Empty function bodies that should have logic
- Stub implementations that return dummy data
- Comments like "implement later" or "add error handling here"
- Skipping edge cases or error handling
- Partial implementations of any kind

### REQUIRED - You must do all of these

- Implement EVERY function completely with real logic

- Handle ALL error cases properly

- Add ALL necessary validation

- Implement ALL edge cases

- Write production-quality code, not prototypes

- If a function needs error handling, ADD IT NOW

- If a function needs validation, ADD IT NOW

- If something needs to be done, DO IT NOW - not later

- Follow package best practices identified during research

- Use recommended patterns from official documentation

## CLEAN CODE PRINCIPLES - Follow These Strictly

Write clean code from the start. Linus Torvalds should approve.

### COMMENTS - Intentional and Rare

Comments exist to explain things the code cannot express on its own. Use them surgically.

**Allowed comments:**

- `// Why` — explains *why* a non-obvious approach was chosen, not what it does:

  ```ts
  // Polling instead of websocket: the admin API rate-limits push events to 1/min
  // so long-polling is cheaper and simpler for this use case
  ```

- **Decision markers** — when a design choice was made deliberately and a future reader might question it, leave a one-liner noting the context:

```ts
// Intentionally not batching: issue #42 requires per-item error isolation
```

- Legal/license headers if required by the project
- Public API documentation (JSDoc/docstrings) for exported library code

**Never add:**

- Comments that restate what the code does (`// increment counter` above `counter++`)
- `// This function does X` — the function name should say that
- Section dividers like `// ========` or `// --- UTILS ---`
- Commented-out code — git remembers
- `// TODO`, `// FIXME`, `// HACK` — implement it now or don't include it

**The code should be self-documenting through clear naming. Comments are the last resort, not the first.**

### DRY - Don't Repeat Yourself

- 2 similar blocks? Extract a function
- Same string used 3+ times? Make it a constant
- Similar conditionals? Consolidate them
- Copy-pasted code with minor variations? Parameterize it
- But don't over-abstract - prefer duplication over the wrong abstraction

### KISS - Keep It Simple, Stupid

- No nested ternaries - use if/else or extract
- Max 4-5 parameters - use an options object if more
- Max 2-3 levels of nesting - extract functions or return early
- No clever one-liners - prefer readable multi-line code
- Delete unused variables, imports, functions, dead code paths

### SOLID Principles

- **Single Responsibility**: One function = one thing. One file = one concern.
- **Open/Closed**: Design for extension, not modification
- **Liskov Substitution**: If inheritance doesn't make sense, use composition
- **Interface Segregation**: Small, focused interfaces
- **Dependency Inversion**: Inject dependencies (but don't over-engineer)

### NAMING - Be Explicit

**Bad → Good:**

- `data`, `info`, `item`, `temp`, `result` → specific names
- `handleClick` → `submitPaymentForm`
- `processData` → `parseUserInput` or `validateOrder`
- `usr` → `user`, `btn` → `button` (no abbreviations)

**Rules:**

- Function names = verbs (`getUser()`, `createOrder()`, `validateInput()`)
- Boolean names = questions (`isValid`, `hasPermission`, `canEdit`)
- No single-letter variables except `i`, `j` in tiny loops
- No Hungarian notation

### ERROR HANDLING

- No empty catch blocks - handle the error or remove try/catch
- No `catch (e) { console.log(e) }` - proper handling or propagation
- Don't swallow errors - let them bubble up or handle properly

### STRUCTURE

- Functions: Max ~50 lines, break up longer ones
- Files: Max ~300-500 lines, split by responsibility
- Use early returns to reduce nesting
- Consistent patterns throughout

## PHASE 4: VERIFICATION LOOP

After implementation, run ALL available verification. This is MANDATORY.

### Auto-detect and run these checks

**JavaScript/TypeScript projects:**

```bash
# Find and run tests
npm test 
# Find and run linting
npm run lint || npx eslint . || npx biome check .
# Find and run type checking
npm run typecheck || npx tsc --noEmit
```

### THE VERIFICATION LOOP

```bash
WHILE any check fails:
    1. Analyze the failure
    2. Fix the code
    3. Re-run the failing check
    4. Continue until it passes
    5. Move to next check
REPEAT until ALL checks pass
```

DO NOT proceed to the next phase until ALL verification passes.

### Post-Verification Commit

Once all checks pass, create a commit for any outstanding verification fixes:

```bash
git add -A
git status --short
# If there are staged changes:
git commit -m "fix(<scope>): resolve verification errors" -m "Refs $ISSUE_REF"
```
